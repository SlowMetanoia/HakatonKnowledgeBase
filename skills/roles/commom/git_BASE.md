## Зачем нужен Git?
Git нужен, чтобы несколько человек могли параллельно писать код, не затирая изменения друг друга, и быстро объединять результат в рабочий прототип.

## Чеклист обучения Git
- Понять, что такое репозиторий и коммит
- Научиться создавать коммиты и просматривать историю
- Освоить работу с ветками
- Понять, как переключаться между ветками
- Научиться сливать изменения

## Что такое Git

Обычно весь этот документ можно было бы ограничить одной ссылкой:
https://learngitbranching.js.org/?locale=ru_RU

Git — это система контроля версий. Проще говоря, мы записываем, как меняется состояние объекта. В нашем случае объект — директория с преимущественно текстовыми файлами.

**Упрощённая модель:**
- Git — это граф коммитов.
- Ветка — это просто указатель на последний коммит.
- ```checkout``` — переключение указателя.
- ```merge``` — добавление нового коммита, объединяющего два состояния.

---

## Инициализация репозитория
Чтобы начать использовать Git в проекте, нужно инициализировать репозиторий:
```bash
git init
```

После этого в директории появляется скрытая папка `.git`, где хранится вся служебная информация.

---

## Коммит
Самая основная команда для создания коммита:
```bash
git commit -m "your commit name"
```

Единицей информации является некоторый набор изменений в существующих файлах — коммит. Обычно коммит зависит от предыдущего, и, поскольку каждый коммит — это изменения, финальная версия объекта — это последовательное применение всех коммитов.

Перед созданием коммита изменения нужно добавить в индекс (staging area):
```bash
git add <file>
```
или сразу все изменения:
```bash
git add .
```

Перед тем, как делать коммит, можно взглянуть на

```bash
git status
```

он отобразит изменённые файлы и подсветит те из них, что добавлены в индекс

## Ветвление, ветки
Изначально в проекте существует основная ветка (обычно `main`).

Ветвление становится актуальным, когда изменения могут вноситься в нескольких направлениях одновременно или над проектом работает несколько человек.

Основная команда для создания ветки:
```bash
git branch <branchname>
```

Эта команда создаёт новую ветку, которая начинается с текущего коммита.

Посмотреть список веток:
```bash
git branch
```

---

## Переключение между ветками
Чтобы перейти на другую ветку:
```bash
git checkout <branchname>
```

Создать ветку и сразу переключиться на неё:
```bash
git checkout -b <branchname>
```

После переключения все новые коммиты будут относиться к текущей ветке.

---

## Слияние веток
Когда работа в ветке закончена, изменения обычно сливаются в основную ветку.

Переключаемся в ветку, куда хотим влить изменения:
```bash
git checkout main
```

Выполняем слияние:
```bash
git merge <branchname>
```

Git попытается объединить изменения автоматически. Иногда возникают конфликты, которые нужно решить вручную.

---

## Удаление веток
После слияния временные ветки обычно удаляют:
```bash
git branch -d <branchname>
```

---
## Работа в команде

Типичный пайплайн для работы в команде выглядит так:
```bash
git pull # вытягиваем изменения из remote
git checkout -b feature-x # создаём новую ветку для внесения изменений и переключаемся на неё
# Далее - выполняем работу (то есть, вносим изменения)
git add -A # добавляем все изменения в индекс
git commit -m "feature-x done: point1, point2, point3" # делаем коммит и описываем, что сделано в комментарии
git checkout main # возвращаемся на main
git pull # обязательно вытягиваем новые изменения из мейна
git merge feature-x # объединяем изменения. Если не получается - разрешаем конфликты
git push
git branch -d feature-x # удаляем временную ветку, если всё прошло успешно
```

---
## Best Practice
- Коммит, не важно, куда он делается, должен описывать одну вещь. Т.е. Если мы переходим на конфиги в новом формате - в коммте должны быть только парсер конфигов, и сущность, которая отвечает за интерпретацию конфигов
- В идеале должна быть отдельная ветка под каждую фичу. То есть, если мы переходим на новые конфиги и одновременно меняем UI, это по хорошему должно происходить в разных ветках. Если это невозможно - мы что-то сделали не так.
- Не работаем в main. В main только сливаем изменения.
- Не изменяем одни и те-же файлы параллельно - чтобы не тратить потом столько-же времени на разруливание конфликтов.

---
## Итог
На базовом уровне Git — это:
- коммиты (фиксация изменений),
- ветки (параллельная работа),
- переключение и слияние.
